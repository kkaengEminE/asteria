# Asteria v0.1.0

## Overview

v0.1.0 is the first official architecture foundation release for Asteria, an extensible AI Publishing OS for operating multiple magazines from shared configuration, prompts, provider boundaries, domains, and workflows.

This release is intentionally mock-first. It proves the system shape without real external API calls, secrets, production publishing, or scheduled automation.

Release verification:

- `npm test` passed with 76 tests.
- `npm run typecheck` passed.
- `npm run dry-run -- "indoor enrichment"` passed.

## Implemented Features

- Minimal TypeScript project setup.
- Magazine config loading and validation.
- Prompt loading, override behavior, validation, and rendering.
- Sequential workflow engine with structured results and failure handling.
- Provider Registry with typed tokens and category-based registration.
- Cat Magazine end-to-end dry run.
- Shared dry-run workflow result shaping.
- CLI dry-run report.
- Disabled GitHub Actions publishing example.
- ADR-based architecture decision log.

## Implemented Providers

- AI:
  - Provider-agnostic AI request, response, usage, message, model, and error contracts.
  - Deterministic Mock AI provider.

- Publisher:
  - Mock-first WordPress publisher adapter draft.

- Image:
  - Mock-first Google Drive image library adapter draft.

- Monetization:
  - Mock-first Coupang affiliate adapter draft.

All providers are dry-run or mock-first. No real API calls are made.

## Workflow

The current Cat Magazine dry run executes:

1. Load Config
2. Load Prompt
3. Research
4. Select Image
5. Generate Article
6. Generate SEO
7. Generate Monetization Preview
8. Publish Preview

The Workflow Engine remains provider-agnostic. Providers are registered and resolved at the composition boundary, then injected into workflow steps.

## Known Limitations

- No real AI provider integration.
- No real article or SEO generation.
- No real WordPress publishing.
- No real Google Drive access.
- No real Coupang affiliate integration.
- No secret management.
- No production scheduler.
- No persistence layer.
- No editorial review UI.
- No retry or rollback policy beyond current workflow failure boundaries.
- Some early `src/core` interfaces overlap with newer domain/provider contracts.
